# ティア設計仕様書

## 概要

5つのティア（Tier 0-4）でドキュメントを階層化し、認知科学に基づいたビジュアル/テキスト比率で最適な学習体験を提供する設計仕様。

### 設計原則

> **10レベルではなく、5つのアクティブレベルで構成された階層化ドキュメントシステムが、包括性とユーザビリティの最適なバランスを実現します。**

Nielsen Norman Groupの研究によると、ユーザーは2〜3段階のProgressive Disclosure（段階的開示）を超えると混乱するが、複雑な技術システムにはより細かい粒度が必要。解決策は、**ユーザーニーズにマッピングされた5つの明確なティア**にコンテンツを整理すること。

---

## 認知科学的基盤

### 作業記憶の制約

認知負荷理論によると、作業記憶は同時に**7±2チャンク**の情報しか処理できない。

| 制約 | 影響 | 対策 |
|------|------|------|
| チャンク制限 | 1セクションに7項目以上は過負荷 | Information Mappingでチャンク化 |
| 専門知識逆転効果 | 初心者向け戦略が専門家には逆効果 | ティア別にアプローチを変える |

### ブルームの分類学

ドキュメントレベルは認知レベルに直接マッピングされる：

| 認知レベル | 対応ティア | 説明 |
|-----------|-----------|------|
| 記憶・理解 | Tier 0-1 | 用語集、概念的概要 |
| 適用 | Tier 2 | ハウツーガイド |
| 分析 | Tier 3 | APIリファレンス |
| 評価・創造 | Tier 4 | カスタマイズ、設計決定 |

213人のコンピュータサイエンス学生を対象とした研究で、各認知レベルが次の順次レベルに大きく影響することが確認された—**レベルをスキップすると学習経路が壊れる**。

### Progressive Disclosure原則

- **具体から抽象へ**: Tier 0は「動作している様子」、Tier 4は「なぜ動作するか」
- **ガイド付きから独立へ**: 低ティアはステップバイステップ、高ティアは参照形式
- **情報の香り**: ユーザーがクリックせずに各ティアの内容を認識できる状態

### スキャフォールディング

教育心理学のスキャフォールディング原則に基づき、ドキュメントは段階的に詳細化する：

```
Tier 0: 全体像を視覚的に把握（60秒）
    ↓
Tier 1: 最初の成功体験（15分）
    ↓
Tier 2: 特定タスクの完了
    ↓
Tier 3: 詳細仕様の参照
    ↓
Tier 4: 設計思想と背景の理解
```

## 5ティア定義

### 概要テーブル

| Tier | 名称 | 時間目標 | V/T比率 | 対象読者 |
|------|------|---------|---------|---------|
| 0 | ビジュアルコンテキスト | 60秒 | 90/10 | エグゼクティブ、新規メンバー |
| 1 | クイックスタート | 15分 | 60/40 | 新規ユーザー |
| 2 | ハウツーガイド | - | 40/60 | 習熟済みユーザー |
| 3 | リファレンス | - | 20/80 | 仕様を確認したいユーザー |
| 4 | 説明・アーキテクチャ | - | 25/75 | アーキテクト、メンテナー |

### Tier 0: ビジュアルコンテキスト

| 項目 | 値 |
|------|-----|
| **時間目標** | 60秒以内 |
| **ビジュアル/テキスト比率** | 90/10 |
| **対象読者** | エグゼクティブ、ステークホルダー、新規メンバー |
| **対応フォルダ** | `03_architecture/context/`, `02_operations/cheatsheets/` |

**必須要素**:
- 単一ページのコンテキスト図
- システム、ユーザー、外部依存関係の可視化
- テキストはラベルと一文の注釈に限定

**判定基準**: A4図上のテキストが読めない場合、情報が多すぎる。

### Tier 1: クイックスタート

| 項目 | 値 |
|------|-----|
| **時間目標** | 15分以内 |
| **ビジュアル/テキスト比率** | 60/40 |
| **対象読者** | 新規ユーザー、学習者 |
| **対応フォルダ** | `01_knowledge/tutorials/` |

**必須要素**:
- 学習目標を冒頭に明記
- スクリーンショット付きの番号付きステップ
- 期待される出力を含むコードスニペット
- 各ステップに検証ポイント
- 所要時間の目安

**判定基準**: 新しいユーザーがこのティアのみでコアワークフローを完了できるか？

**ミニマリズム原則**（John Carroll）:
1. アクション指向アプローチ
2. タスクドメインへの固定
3. エラー回復サポート
4. モジュール式読書

### Tier 2: ハウツーガイド

| 項目 | 値 |
|------|-----|
| **時間目標** | タスク依存 |
| **ビジュアル/テキスト比率** | 40/60 |
| **対象読者** | 基本習熟済みのユーザー |
| **対応フォルダ** | `01_knowledge/03-how-to/`, `02_operations/` |

**必須要素**:
- 前提条件を明確に
- 1ドキュメント1タスク
- 結果の確認方法
- 一般的なエラーのトラブルシューティング
- 分岐ワークフローのデシジョンツリー

**前提**: Tier 1（クイックスタート）完了

### Tier 3: リファレンス

| 項目 | 値 |
|------|-----|
| **時間目標** | ルックアップ |
| **ビジュアル/テキスト比率** | 20/80 |
| **対象読者** | 仕様を確認したいユーザー |
| **対応フォルダ** | `01_knowledge/reference/` |

**必須要素**:
- 網羅性を重視
- 構造化された形式（テーブル、定義リスト）
- 検索しやすいキーワード配置
- バージョン情報必須

**最適化**: 線形読書ではなく、検索とスキャンに最適化。

### Tier 4: 説明・アーキテクチャ

| 項目 | 値 |
|------|-----|
| **時間目標** | 深い理解 |
| **ビジュアル/テキスト比率** | 25/75 |
| **対象読者** | アーキテクト、メンテナー、シニア開発者 |
| **対応フォルダ** | `01_knowledge/01-concepts/`, `03_architecture/` |

**必須要素**:
- アーキテクチャ決定記録（ADR）
- 設計トレードオフ分析
- コンポーネント相互作用図
- 「なぜ」の説明

**更新頻度**: 下位ティアより変更頻度が低い—それに応じて投資。

### フレームワーク統合

**Diátaxis**:
- チュートリアル → Tier 1
- ハウツー → Tier 2
- リファレンス → Tier 3
- 説明 → Tier 4

**C4モデル**:
- Context（Level 1）→ Tier 0
- Containers（Level 2）→ Tier 4
- Components（Level 3）→ Tier 4

**運用ドキュメント階層**:
- プロセス/プレイブック/ランブック → Tier 2
- チートシート → Tier 0（ビジュアル重視）またはTier 3（テキスト重視）

## ビジュアル/テキスト比率ガイドライン

### 比率の根拠

研究により、脳は画像を**テキストより60,000倍速く**処理し、ユーザーはビジュアル指示で**67%成功率が向上**することが確認されている。

### ティア別ガイドライン

| ティア | 比率 | ビジュアル要素 | テキスト要素 |
|--------|------|--------------|------------|
| 0 | 90/10 | コンテキスト図、アーキテクチャ概要 | ラベル、一文注釈 |
| 1 | 60/40 | スクリーンショット、ワークフロー図 | 手順テキスト |
| 2 | 40/60 | デシジョンツリー、フローチャート | 手順、トラブルシューティング |
| 3 | 20/80 | ER図、シーケンス図 | テーブル、定義リスト |
| 4 | 25/75 | アーキテクチャ図、相互作用図 | 設計説明、トレードオフ分析 |

### Diagram-as-Code推奨

| ツール | 用途 | 利点 |
|--------|------|------|
| Mermaid | シンプルな図 | GitHub Markdownネイティブ対応 |
| PlantUML | C4モデル | 詳細なアーキテクチャ図 |

## CFD/数値シミュレーションへの適用例

### ティア別コンテンツマッピング

| ティア | CFD/数値シミュレーションでの内容 |
|--------|--------------------------------|
| 0 | シミュレーションパイプライン全体図（前処理→ソルバー→後処理） |
| 1 | 簡単なケースを実行するクイックスタート（例：2D流れ解析） |
| 2 | メッシュ生成、ソルバー設定、後処理タスクのワークフローガイド |
| 3 | ソルバーパラメータ、境界条件、物性値のリファレンス |
| 4 | 数値手法の説明、安定性・収束理論、スキーム選択の根拠 |

### 実装例：ROMS海洋シミュレーション

```
Tier 0: ROMS全体アーキテクチャ図（60秒で把握）
    ↓
Tier 1: 黒潮シミュレーション入門チュートリアル（15分で初回成功）
    ↓
Tier 2: メッシュ生成手順、境界条件設定手順
    ↓
Tier 3: ROMSパラメータリファレンス（名前、型、デフォルト値、説明）
    ↓
Tier 4: 数値スキームの選択理由、収束条件の設計思想
```
